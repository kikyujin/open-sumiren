# SUMIRE Notebook コンセプトドキュメント v3

**作成日**: 2026-03-20
**改訂日**: 2026-03-22
**作成者**: エルマー🦊 + マスター
**ステータス**: Phase 0 実証済み・コンセプト整理

---

> **「これ持ってたら、すごいことでもろくでもないことでも全部管理できちゃう魔法のアプリ」**

大学生が `勉強 → 就活 → バイト` を管理するのも、
CEOが `HANS → xMBS → ET1` を管理するのも、
どっかの国の大統領が `外交 → 経済 → 軍事` を管理するのも、
構造は全部同じ。

テーブルがあって、セルがあって、参照があって、要約が上に流れてくる。
違うのは中身の深さと複雑さだけ。

Excelが全人類の「計算」を民主化したのと同じ構造で、
**SUMIRE Notebookは全人類の「あれどうなってたっけ」を民主化する。**

Excelのセルに `=SUM()` って書いたら計算してくれるのと同じノリで、
SUMIRE Notebookのセルに `=LLM("このgitリポジトリを要約して")` って書いたらLLMが読んで返してくれる。

**Excel = 数値の関数が入ったセル。SUMIRE Notebook = LLMが入ったセル。**

使う側はLLMの仕組みなんか知らなくていい。セルに書くだけ。
おばちゃんが `=SUM` の中身わかんなくても使えるのと同じ。

## 1. 何を解決するか

### 問題

マスターは多数のプロジェクトを並行で走らせている。
会社の業務（HANS、ファイナンス、大連・台北スタジオ）から、
創作活動（xMBS、CAMERA、DAN-SHBOARD、ElmarTail One）まで、
性質も深さもバラバラなプロジェクトが同時に存在する。

各プロジェクトのドキュメントはObsidianやgitに存在するが、
**「あれどうなってたっけ？」→ 自分でファイルを探す → コンテキストに入れる → LLMに聞く**
という作業を毎回マスターがやっている。これがボトルネック。

### 既存ツールの限界

- **Claude Code / OpenClaw / Devin 系**: シーケンシャル実行は得意。だが**俯瞰できない**。目の前のタスクをこなす作業者であって、全体を把握する管理者ではない
- **Obsidian**: ドキュメントはあるが、プロジェクト横断の俯瞰ビューがない
- **CLAUDE.md**: プロジェクト単位の文脈は持てるが、プロジェクト間の関係や全体像は持てない
- **アウトライナー/ツリー型**: 整理はできるが、結局リニア。求める「表」の感覚ではない

## 2. SUMIRE Notebook とは

**LLMをセルの計算エンジンとして持つスプレッドシート。**

Univerをベースに、セルの中で `=LLM(...)` と書けばLLMが動く。Excelユーザーが `=SUM()` を使う感覚で、LLMに「要約して」「指示書を作って」と頼める。

### Excelとの対比

| Excel | SUMIRE Notebook |
|-------|-----------------|
| セルに数値・数式が入る | セルに値・LLM式・ファイルパスが入る |
| `=SUM(A1:A10)` | `=LLM(A1, "要約して", B1)` |
| セル参照で計算が連鎖 | セル参照でLLM処理が連鎖（将来） |
| 再計算は即時 | 再計算はオンデマンド（▶ ボタン） |
| シートは1ファイル | ノートブックは1ファイル。ノートブック間リンクで拡張 |

### スケーラビリティ

Univerのシートは列が A〜T（20列）、行が 1〜999。単一ノートブックとしては実質十分な広さがある。

プロジェクトが増えてシートに収まらなくなったときは、**別の SUMIRE Notebook とのリンク**で拡張する。各ノートブックが独立したファイル（JSONスナップショット）で、ノートブック間のセル参照で情報を繋ぐ。Excelの「別ブック参照」と同じ発想。

```
[全体管理.sumiren]          [xMBS.sumiren]             [CAMERA.sumiren]
┌──────────────────┐       ┌──────────────────┐       ┌──────────────────┐
│ プロジェクト  状態 │       │ タスク    状態    │       │ Phase   状態     │
│ xMBS      →参照  │──────→│ セルAPI   完了    │       │ Phase1  稼働中   │
│ CAMERA    →参照  │──┐    │ テンプレ  実装中  │       │ Phase2  設計中   │
│ DAN-SH    稼働中  │  │    │ 次: =LLM(...)    │       └──────────────────┘
└──────────────────┘  │    └──────────────────┘              ▲
                      └──────────────────────────────────────┘
```

巨大な1シートではなく、プロジェクト単位のノートブックを繋ぐ。これにより：
- 各ノートブックが軽量に保てる
- プロジェクトの追加・削除がファイル単位で完結する
- ちびエルマーに渡す単位としても扱いやすい

## 3. コア概念

### 3.1 セルの中身

| 種類 | 例 | 説明 |
|------|-----|------|
| 手書き値 | `v0.5.3` `頑張れ` | 人間が書いた固定値 |
| ファイルパス | `file:///Users/mahito/StockDB/xMBS.md` | ローカルファイルへの参照。LLM実行時に内容が自動展開される |
| LLM式 | `=LLM(B2, "要約して", D2)` | 入力セル・プロンプト・出力先を指定してLLMを実行 |

### 3.2 LLM式の文法

```
=LLM(input, "prompt", output)
```

| 引数 | 必須 | 説明 |
|------|------|------|
| input | 任意 | 入力データのセル参照。値がファイルパスなら自動でファイル内容に展開 |
| "prompt" | 必須 | LLMへの指示。ダブルクォーテーションで囲む |
| output | 任意 | 結果の書き込み先セル。省略時は数式セル自体に上書き（数式が消える） |

**例:**
```
=LLM(B2, "1行に要約して", D2)    — B2の内容を要約してD2に出力
=LLM(B2, "英語に翻訳して", D2)   — B2の内容を翻訳してD2に出力
=LLM("今日は何曜日？")            — プロンプトのみ（入力なし）
```

出力先セルを指定すると、数式セルには `=LLM(...)` が残り、何度でも再実行できる。

### 3.3 関数ファミリー

| 関数 | エンジン | 用途 |
|------|---------|------|
| `LLM` | Ollama（ローカル） | 通常のLLM処理。gemma3:27b等 |
| `xLLM` | Claude API等（クラウド） | 高品質な処理が必要なとき。APIキーはバックエンドで管理 |
| `ClaudeCode` | Claude Code | コード生成・実行をClaude Codeに委譲。ちびエルマーへの指示書生成等 |
| `Launch` | OS標準 | ファイルを拡張子に紐づけられたアプリで開く。Typora、VSCode等 |

全ての関数が同じ文法に従う：
```
=関数名(input, "prompt/path", output)
```

**Launch の例:**
```
=Launch(B2)                        — B2のファイルパスをOSのデフォルトアプリで開く
=Launch("file:///Users/mahito/doc.md")  — 直接パス指定で開く
```

Launchは結果を返さない（出力先不要）。「開いた」ことのログだけ返す。

### 3.4 セルの状態

| 状態 | 意味 | Excelで言うと |
|------|------|-------------|
| 値あり | 正常。手書き値またはLLM結果が入っている | `42` |
| `#UNREF` | 参照先のセルが存在しない / ファイルが見つからない | `#REF!` |
| `#UNSOLVED` | LLM式がまだ実行されていない / 実行に失敗した | `#VALUE!` |
| `#STALE` | キャッシュはあるが参照先が更新されている | Excelにはない新概念（将来実装） |

**システムは状態を見せるだけ。再計算のトリガーは人間。**
Excelが `#REF!` を勝手に直さないのと同じ。

### 3.5 ファイルパスの自動展開

セルに `file:///` または `/` で始まるパスが入っている場合、LLM実行時にバックエンド（`/api/read`）経由でファイル内容に自動展開される。

- UTF-8テキストのみ対応
- 100KB上限（超過時は切り詰め＋警告）
- セルの表示はパス文字列のまま（展開はLLM実行時のみ）

## 4. 使い方のフロー

### 「あれどうなってたっけ？」のとき

1. SUMIRE Notebookを開く
2. ファイルパスのセルを確認（`file:///...`）
3. 隣のセルに `=LLM(B2, "現在の状態を教えて", D2)` と書く
4. ▶ ボタンを押す → ファイルが読み込まれ → LLMが要約 → D2に結果
5. 結果を見て次のアクションを決める

### 新しいプロジェクトを追加するとき

1. セルにプロジェクト名を書く
2. 隣に仕様書やREADMEのファイルパスを書く
3. `=LLM(B5, "次のタスクは？", D5)` で状態を要約
4. Ctrl+S で保存

### ちびエルマーに作業を渡すとき

1. `=LLM(B2, "ちびエルマー指示書を作成して", D2)` を実行
2. D2に指示書が生成される
3. `=Launch(D2)` で指示書をエディタで開く（将来）
4. ちびエルマーにコピペで渡す（将来的には `=ClaudeCode(D2)` で直接渡す）

## 5. 管轄

**スミレ管轄。** 理由：

- 会社の業務も創作もカバーする（秘書として全体把握が本業）
- 「あれどうなってたっけ」に答えるのは秘書の仕事
- ファイナンスも進捗もプロジェクトも、全部スミレのデスクを通る
- 実働（ファイル操作、サービス実行）はダンチャンやちびエルマーに委譲

## 7. 実装アプローチ

### 7.1 採用: Univer + 数式エンジンの外でLLM実行

**Univerをスプレッドシートの表示・編集層として使い、LLM実行は数式エンジンの外で行う。**

| 層 | 担当 | 技術 |
|----|------|------|
| セル表示・編集・IME | Univer (canvas) | @univerjs/preset-sheets-core |
| セル参照追従 | Univerの数式エンジン | ダミーLLM関数を登録 |
| LLM実行トリガー | カスタムUI | ▶ ボタン / 右クリック / Ctrl+Enter |
| LLM実行・ファイル読み取り | FastAPIバックエンド | /api/llm, /api/read |
| LLMエンジン（ローカル） | Ollama | gemma3:27b |
| LLMエンジン（クラウド） | Claude API等 | xLLM関数用 |
| 保存・復元 | Univerスナップショット + FastAPI | /api/save, /api/load |

### 7.2 ダミーLLM関数（参照追従用）

Univerの数式エンジンに `LLM` をカスタム関数として登録するが、**実際のLLM呼び出しはしない**。常に空文字を返す。

これにより：
- `=LLM(B2, "要約", D2)` が正式な数式としてUniverに認識される
- B2やD2を参照しているため、行挿入・コピペ時に参照が自動追従する
- 数式エンジンの自動再計算が走っても空文字が返るだけで副作用がない
- 実際のLLM呼び出しは ▶ ボタン経由で、数式エンジンとは独立して実行される

### 7.3 出力先セル方式（案B）

LLM実行結果を数式セルに直接書き込むと数式が消えてしまう（Univer v0.18.0の制約: 数式エンジンの再計算が即座に表示値を上書きする）。

解決策として、`=LLM(input, "prompt", output)` の第3引数で出力先セルを指定する。数式セルは触らず、結果は指定セルに書き込まれる。これにより数式が保持され、何度でも再実行可能。

### 7.4 検討済みで不採用としたアプローチ

| アプローチ | 不採用理由 |
|-----------|-----------|
| React自作セルエディタ | IME（日本語入力）が構造的に壊れる。Reactの英語圏前提設計に起因 |
| React Flow グラフ型 | IME回避の迂回策。複雑化しただけで本質的解決にならず |
| Univerの数式エンジンにLLM直載せ | 非同期バグ、再計算タイミング制御不能、並列実行問題 |
| LLM実行をサイドパネルに分離 | 「セルの中で動く」直感性が死ぬ。ダッシュボードになる |
| 数式と値の同時セット（案A） | Univer v0.18.0で再計算が即座にvを上書き。構造的に不可 |
| Excel/LibreOffice + カスタム関数 | ファイル参照・git連携がセルの器に収まらない |
| Google Apps Script | ローカルLLMに繋がらない。実行時間制限 |
| SwiftUI ネイティブ | Linuxに持っていけない。OSS化と矛盾 |
| Handsontable | 非商用ライセンス。IME対応もバグあり |
| AG Grid | IME対応不明。スプレッドシート的UXが弱い |
